The following page provides a simple example about how the test framework has assisted to improve the experience for a codec developers.
The case has been run on the SVT-AV1 whose codebase could found from github at SVT-AV1 and with a pre-configuration whose details you could found at the page Configuration.

The dashboard of test framework
The default dashboard of test framework has been made up of three tests and a regression report section which is specially designed to analyze the testing results generated by various testing tools.
Coverage report
This testing tool has been designed to find out to which extent the codec has been covered by the test suit deployed.
The result shall be displayed in a percentage which stands for how much the codec has been covered by the test suits, the higher the more parts of the codec is covered.
This time the test has told us that 92.4% of our codec has been covered by the test suits which means XXXX functions out of XXXXX functions and XXXXX lines out of XXXXX lines could be tested each time a test is run.
Conformance report
With this test, users could confirm if the encoded bit streams comply with a specialized codec specifications.
The result is generated by running a certain number of tests on the codec that has been provided, to find out how many tests they've passed. The more tests the codec has passed, the more steady the codec is.
The result is also provided as a percentage while the higher that number is the less crashes or discrepancies shall happen to the codecs.
This time the codec has passed all tests which means this is a highly steady version.
Performance report
This tool will provide some important data points that could help users to analyze a codec's performance, including but not limited to BD-rate, PSNR, VMAF and speed related variables.

Regression Report
The regression report is a centre where we can further analyze and find out all regressions happened in tests that have been run on the framework.
As we could see an alert under the performance test, there is a regression happened on the commit ID of XXXXX:
with a single click on the alert, we could find details of the regression as below, yet we have no idea what changes we've made that has caused this regression. In order to find out, we could run a further analysis on all those commits that have been submitted and find out the exact one that has caused this.
The details of the analysis could be found at regression report, as for this time we could see that the commit ID of XXXXXX has caused this regression.
With a modification and another test we confirmed that the regression has been fixed and a comment could be added to the regression specifying how this has been fixed. Hopefully this could be a reference for any similar issues that'll happen in future.
